城市故障分析方法論

把城市問題當成一個可以被觀察、分類、追蹤與改善的系統。這裡說明我們怎麼看資料、怎麼分類問題、怎麼判讀熱點，以及 prototype data 目前有哪些限制。

方法論的任務

不是把每件事都說成高深模型，而是讓一般人知道：這個網站怎麼分類問題、怎麼看熱點、為什麼不直接把 prototype data 當成正式結論。

方法論重點：目前是 prototype dashboard，不是正式全量資料。所有數字與熱點都應清楚標示來源、限制與更新狀態；正式資料接入前，維持 no live crawler、no source_url requests、no personal data、no auto publish。

什麼是城市故障分析

白話來說，就是把城市裡反覆出現、會影響生活品質的問題，整理成可以追蹤的分類與地點，而不是只靠零散抱怨或單次新聞。

plain language

看問題類型

先把交通、停車、環境、通學安全、市場商圈等議題分類清楚，避免不同問題混在一起。

看問題位置

再看哪些地點反覆出現類似問題，判斷是否形成城市熱點。

看可執行行動

最後把熱點對應到主管機關與改善方向，讓資料能變成實際討論的起點。

怎麼分類問題

分類不是為了學術好看，而是為了讓不同來源、不同時間的問題可以放到同一個結構裡比較。

issue taxonomy

目前主要題組

  • 交通與停車
  • 道路與人行空間
  • 環境衛生
  • 通學安全
  • 市場商圈與動線

分類的用途

  • 讓首頁可以做議題排行。
  • 讓地圖可以對應熱點類型。
  • 讓後續正式資料接入時，欄位可以延續同一套分類。

怎麼看熱點

熱點不只是某個地方很有名，而是某個區域反覆出現多種城市問題，值得優先觀察。

hotspot logic

1. 看位置

同一地點若反覆出現交通、停車、環境或安全問題，就可能形成城市熱點。

2. 看問題強度

目前用 prototype score 表示熱點強度，正式版會再接入更多資料欄位與時間序列。

3. 看可執行行動

熱點不是只做排名，還要對應主管機關、改善方向與後續追蹤。

prototype data 的限制

prototype data 的目的，是先讓網站可展示、可解釋、可討論；但不能假裝它已經是正式資料庫。

data limitation

目前能做什麼

  • 展示儀表板樣貌。
  • 呈現原型 KPI、議題排行與熱點。
  • 說明後續要接哪些正式資料。

目前不能說什麼

  • 不能說這是正式全量資料。
  • 不能拿 prototype data 當正式民意或完整統計。
  • 不能把未確認來源標成 approved_for_crawling。

後續如何接正式資料

正式資料接入不是直接抓回來就好，而是要先確認來源、欄位、限制與更新方式。

next data

確認官方來源

先確認資料來源、欄位、更新頻率與限制。

建立資料欄位

把時間、地點、問題類型、主管機關與處理狀態整理成可分析欄位。

更新儀表板

讓首頁、地圖、總控台從 prototype data 逐步接入正式資料。
