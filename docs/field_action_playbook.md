# 城市故障現場行動手冊 v1

本文件定義如何把 dashboard 上的城市故障資料，轉成現場走訪、里鄰拜訪、拍攝紀錄與改善追蹤。

---

## 目的

資料站不只是展示數字，而是要協助把問題轉成可行動的地方工作流程：

```text
資料發現
→ 現場確認
→ 拍照 / 影片紀錄
→ 議題摘要
→ 局處或地方溝通
→ 改善追蹤
→ 回到週報更新
```

---

## 一、如何選擇現場走訪地點

優先順序：

1. 城市故障分數高。
2. 近期趨勢上升。
3. 多個資料來源都出現。
4. 涉及交通、人行、通學安全或公共安全。
5. 適合用照片或影片說明。
6. 可提出具體改善建議。

---

## 二、現場查核前準備

出發前應準備：

- 熱點名稱
- 地點描述
- 主要議題
- 相關資料來源
- 已知問題摘要
- 可能相關局處
- 拍攝重點
- 安全注意事項

建議格式：

```text
熱點：文化路商圈周邊
議題：停車 / 人行 / 商圈動線
觀察時間：平日傍晚 / 假日晚上
拍攝重點：臨停、人行動線、卸貨區、行人穿越
注意事項：不拍攝可識別個人正臉、不拍車牌特寫
```

## 三、現場紀錄欄位

```json
{
  "visit_id": "field-2026-0001",
  "visited_at": "2026-05-21 18:30:00",
  "location_name": "文化路商圈周邊",
  "district": "西區",
  "primary_issue": "traffic",
  "secondary_issues": ["pedestrian", "market"],
  "observations": [
    "尖峰時段臨停車輛增加",
    "部分行人被迫走到車道邊緣",
    "商圈卸貨與行人動線重疊"
  ],
  "photo_count": 0,
  "video_count": 0,
  "recommended_action": "盤點臨停熱點與行人穿越動線，評估卸貨時段與行人安全改善。",
  "privacy_checked": true
}
```

---

## 四、拍攝注意事項

可以拍：

- 道路狀況
- 路口動線
- 人行道障礙
- 公共設施
- 標線與號誌
- 整體場景

避免拍：

- 可識別正臉
- 車牌特寫
- 私人住家門牌
- 未經同意的個人對話
- 孩童特寫
- 醫療或敏感個人資訊

---

## 五、現場訪談原則

若與市民聊天，應避免誘導式問題。

建議問法：

```text
這附近平常最常遇到什麼交通或人行問題？
這個問題大概什麼時間最明顯？
你覺得最需要先改善的是什麼？
```

避免問法：

```text
你是不是也覺得某局處很爛？
你是不是不滿現任民代？
你會支持誰？
```

## 六、現場資料回寫流程

```text
現場紀錄
→ 去識別化
→ 議題分類
→ 地點標準化
→ 更新 hotspot notes
→ 週報引用
→ 社群素材草稿
```

---

## 七、改善追蹤欄位

```json
{
  "tracking_id": "track-2026-0001",
  "issue": "文化路商圈周邊人行安全",
  "status": "observed",
  "next_step": "整理照片與動線圖，提出改善建議",
  "responsible_unit": "待確認",
  "last_checked_at": "2026-05-21",
  "next_check_at": "2026-06-01"
}
```

status 建議：

| status | 說明 |
|---|---|
| observed | 已現場觀察 |
| documented | 已整理資料 |
| submitted | 已送出建議或陳情 |
| responded | 已收到回應 |
| improved | 已改善 |
| monitoring | 持續觀察 |

---

## 八、從現場轉成社群內容

每個現場議題可轉成：

1. 一張圖卡
2. 一篇 FB 長文
3. 一則 Threads 短文
4. 一段 30 秒短影音
5. 一個週報段落
6. 一筆 dashboard 更新

---

## 九、原則

1. 不用情緒取代證據。
2. 不用單一案例代表全部。
3. 不拍攝或公開個資。
4. 不攻擊個人。
5. 把抱怨轉成可追蹤的改善清單。
6. 每次現場走訪都要能回到資料與行動。
7. 地名應符合嘉義在地稱呼，參考 `docs/local_terminology_style_guide.md`。
