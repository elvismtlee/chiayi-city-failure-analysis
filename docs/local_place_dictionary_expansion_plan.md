# 嘉義在地地名字典擴充計畫 v1

本文件規劃如何逐步擴充 `dashboard/data/local_place_dictionary.json`，避免一次加入大量未確認地名，造成 dashboard、週報或 AI 摘要使用錯誤稱呼。

---

## 目標

建立一份可被 dashboard、資料處理流程、AI 摘要與週報共同使用的嘉義在地地名字典。

地名字典應同時支援：

- 對外顯示名稱
- 嘉義人日常稱呼
- 正式名稱
- 可接受別名
- 不建議用語
- 常見議題標籤
- 地理精度
- 人工確認狀態

---

## 擴充原則

1. 先放高頻公共地點，不先放私人地址。
2. 先放西區，再逐步擴到全市。
3. 先放地標、商圈、市場、交通節點，再放細路段。
4. 地名不確定時，不直接進正式 JSON，先放待確認清單。
5. 新增地名後必須跑地名字典測試。
6. 若嘉義在地說法與觀光說法不同，優先使用在地說法。

---

## 第一階段：高頻公共地點

建議先確認並擴充：

| 類型 | 例子 | 備註 |
|---|---|---|
| 商圈 | 文化路商圈 | 已加入 |
| 車站 | 嘉義火車站前站、嘉義火車站後站 | 前站 / 後站建議分開 |
| 市場 | 北興市場周邊、其他市場周邊 | 需人工補完整清單 |
| 公共空間 | 中央廣場周邊 | 已加入 |
| 學校周邊 | 學校周邊通學路線 | 若不指涉單一學校，可先用通稱 |
| 主要道路 | 民生南路、新民路、世賢路 | 建議寫成「部分路段」 |
| 公園 | 公園周邊 | 需人工補具體公園名 |

---

## 第二階段：西區生活熱點

建議由人工提供或確認：

```text
市場清單
學校清單
公園清單
重要路口
停車熱點
通學路線
常被反映的路段
```

每個地點確認時，至少要有：

```text
display_name:
local_name:
place_type:
district:
aliases:
avoid_terms:
issue_tags:
人工確認人:
確認日期:
```

---

## 第三階段：加入機器可讀 JSON

確認後才加入：

```text
dashboard/data/local_place_dictionary.json
```

新增後應跑：

```bash
pytest tests/test_local_place_dictionary.py
pytest tests/test_local_terminology.py
```

## 待確認地名格式

尚未確認的地名不要直接放進正式 JSON，可先用下列格式整理：

```text
原始地名：
建議顯示名稱：
嘉義人口語稱呼：
正式名稱：
類型：
行政區：
常見議題：
不建議用語：
是否已人工確認：否
備註：
```

## 命名注意事項

### 路段

若不是完整道路，不要只寫整條路。

建議：

```text
民生南路部分路段
新民路部分路段
世賢路部分路段
```

### 學校

若不確定是否要指涉單一學校，先使用：

```text
學校周邊通學路線
```

### 市場

市場議題通常要加上「周邊」，因為問題常發生在市場外圍道路、停車、卸貨或行人動線。

建議：

```text
北興市場周邊
市場周邊
```

### 商圈

商圈不要一律稱為夜市或觀光區。

建議：

```text
文化路
文化路商圈
文化路商圈周邊
```

## 人工確認原則

地名字典擴充前，應由熟悉嘉義在地生活的人確認：

1. 這個名稱嘉義人平常會不會這樣說？
2. 這個名稱是否太像觀光文案？
3. 對外顯示會不會讓市民覺得怪？
4. 是否需要區分前站 / 後站、東側 / 西側、路段 / 路口？
5. 是否有容易誤用的名稱要放入 avoid_terms？

---

## 原則

```text
地名字典要慢慢擴充，寧可少而準，不要多而錯。
```
