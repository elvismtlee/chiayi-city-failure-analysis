# 一人團隊操作流程｜競選資料總控台

更新時間：2026-05-23 01:26 Asia/Taipei

本文件說明一人團隊如何使用 `dashboard/` 內部工作台。這套 dashboard 是內部資料整理與人工審核工具，不是民調、不是支持度調查，也不會自動發布任何內容。

---

## 一、核心原則

1. 先整理公開資料，再做人工審核。
2. 先確認來源，再形成結論。
3. 先進入審核佇列，再進入可用素材。
4. 所有對外素材都要人工查證、人工審核、人工手動發布。
5. 不得放入個資、私人陳情、捐款人、志工名單、電話、地址或未公開內部策略。
6. 所有政策與社群內容都要保留「初步觀察、待查證、公開資料整理」的保守語氣。

---

## 二、每日使用順序

### 1. 先看健康檢查

頁面：`dashboard/health-check.html`

用途：確認 dashboard 需要的 JSON、HTML、JS 與導覽入口是否完整。

處理方式：

- 若 status 是 `ok` 或 `ready_for_manual_review`：可以進入人工審核流程。
- 若 status 是 `needs_attention`：先看 warnings，補缺檔、修 JSON 或補導覽。
- 若有 invalid JSON：先修資料檔，不要繼續產生對外素材。

### 2. 再看總控台

頁面：`dashboard/command-center.html`

用途：快速掌握資料流、審核 backlog、下一步工作與 warning。

每日優先看：

- 哪些資料來源 available。
- 哪些 queue 還沒人工審核。
- 哪些 public review items 需要確認。
- 哪些素材已經 approved 但尚未手動發布。

### 3. 進入資料審核頁

依序檢查：

| 頁面 | 任務 |
|---|---|
| `minutes-review.html` | 檢查會議紀錄解析結果是否合理。 |
| `minutes-issues.html` | 檢查議題候選是否有來源與上下文。 |
| `weekly-summary.html` | 檢查每週摘要是否適合內部使用。 |
| `policy-drafts.html` | 檢查政策草稿是否具體、可執行、無誇大。 |
| `social-drafts.html` | 檢查社群草稿是否正面、可查證、無攻擊。 |
| `video-scripts.html` | 檢查短影音腳本是否容易理解、無未查證指控。 |
| `filming-checklists.html` | 檢查拍攝地點、畫面、安全與法遵提醒。 |

### 4. 檢查公開審核佇列

頁面：`dashboard/public-review.html`

用途：放所有準備對外使用，但尚未人工核准的素材。

判斷原則：

- `risk_level` 高：必須人工查來源與文字風險。
- `evidence_status` 不完整：不可公開。
- `review_status` 不是 approved：不可公開。
- 對人、機關、廠商或工程品質有具體指稱時，要回到原始資料確認。

### 5. 檢查已核准素材

頁面：`dashboard/approved-materials.html`

用途：只放已經人工審核通過的素材樣本。

注意：

- approved 不等於自動發布。
- 發布前仍需再次確認日期、語氣、圖片與法遵。
- 若後來發現來源有變動，要移回 public review queue。

## 三、每週使用順序

### 1. 產生每週系統報告

頁面：`dashboard/weekly-system-report.html`

用途：彙整總控台、健康檢查、公開審核與已核准素材。

每週要看：

- 本週可用資料量。
- 本週缺漏資料。
- 本週應優先審核項目。
- 本週可轉成政見、社群、短影音的素材。

### 2. 產出一週行動

建議每週產出：

| 輸出 | 數量 |
|---|---:|
| 地方議題摘要 | 10 個 |
| 政策草稿 | 3 則 |
| 社群貼文草稿 | 5 則 |
| 短影音腳本 | 3 支 |
| 拜訪或拍攝地點 | 5 個 |

---

## 四、資料狀態定義

| 狀態 | 意義 | 是否可公開 |
|---|---|---|
| draft | 系統或人工草稿 | 不可公開 |
| needs_review | 等待人工審核 | 不可公開 |
| needs_evidence | 缺來源或上下文 | 不可公開 |
| approved | 已人工核准 | 可人工手動發布 |
| archived | 保留紀錄，不再使用 | 不公開 |

---

## 五、發布前檢查清單

任何內容要放到 Facebook、Threads、LINE、YouTube、官網或文宣前，至少確認：

1. 有公開來源或可說明的地方觀察。
2. 沒有私人個資。
3. 沒有未查證指控。
4. 沒有人身攻擊。
5. 沒有把草稿說成正式結論。
6. 沒有把 dashboard 數據說成民調或支持度。
7. 語氣正面、具體、可執行。
8. 結尾有合適的行動呼籲。

---

## 六、建議的一天工作節奏

| 時段 | 任務 | 預估時間 |
|---|---|---:|
| 上午 | 看 health check 與 command center | 10 分鐘 |
| 上午 | 審核 3 筆地方議題或會議紀錄 | 30 分鐘 |
| 下午 | 整理 1 則政策草稿與 1 則社群貼文 | 40 分鐘 |
| 傍晚 | 選 1 個可拍短影音題目 | 20 分鐘 |
| 晚上 | 把已核准素材加入隔日手動發布清單 | 20 分鐘 |

## 七、安全邊界

這個 repo 若維持 public，請只放：

- 公開資料。
- fixture。
- sample。
- 已去識別化資料。
- 內部流程文件。
- 不含個資的草稿樣本。

不要放：

- 捐款人資料。
- 志工名單。
- 支持者名單。
- 私人陳情內容。
- 個人電話、地址、身分證字號。
- 未公開競選策略或敏感時程。
- 帳號、密碼、token、API key。

---

## 八、給使用者的簡單判斷

當 dashboard 出現一個素材時，先問三句：

```text
這是真的公開資料嗎？
這已經人工審核了嗎？
這樣講，市民會覺得清楚、正面、可信嗎？
```

三個答案都是「是」，才適合手動對外使用。
